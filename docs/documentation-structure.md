# Структура документации

## Назначение

Этот документ задает обязательную структуру документации проекта.

Цель:

- сделать документацию предсказуемой
- разделить продуктовые, архитектурные и валидационные вопросы
- описывать каждую фичу по одному и тому же каркасу

## Главные оси документации

Любая значимая фича, flow или подсистема должны описываться по трем осям:

1. Что строим
2. Как строим
3. Как проверяем

Это обязательное разделение. Не допускается смешивать все в один неструктурный
текст.

## Ось 1. Что строим

Эта часть отвечает на продуктовые вопросы.

Обязательные темы:

- какую проблему решаем
- для кого это решение
- какой результат нужен
- что входит в scope
- что не входит в scope
- какие ограничения или предпосылки есть

Эта часть не должна проваливаться в детали реализации.

## Ось 2. Как строим

Эта часть отвечает на инженерные вопросы.

Обязательные темы:

- архитектура решения
- данные и состояния
- интерфейсы и интеграции
- ключевые технические решения
- конфигурация
- ограничения реализации

Именно здесь фиксируются статусы, переходы, контракты, схемы конфигов, runtime
model, взаимодействие с GitHub, zellij и другими подсистемами.

## Ось 3. Как проверяем

Эта часть отвечает на вопросы корректности и готовности.

Обязательные темы:

- критерии корректности
- критерии готовности
- инварианты
- сценарии проверки
- тестовые случаи
- наблюдаемость и диагностика

Если фича не имеет явных критериев проверки, она считается недоописанной.

## Уровни документации

Документация в репозитории делится на уровни:

Документация должна раскрываться по принципу `progressive disclosure`: от
краткой входной точки к более детализированным документам через явные ссылки.

### 1. Уровень проекта

Документы этого уровня описывают весь продукт целиком.

Примеры:

- `README.md`
- архитектурные обзоры
- правила документации
- правила для `Плана имплементации`

Для уровня проекта действует отдельное правило для `README.md`:

- `README.md` является repo-level входной точкой и верхнеуровневым обзором
  продукта
- `README.md` может быть каноническим источником только для знаний уровня
  всего репозитория: назначения продукта, границ MVP, общей карты документации
  и краткой архитектурной рамки
- `README.md` не должен подменять SSOT, feature-документы, ADR и
  verification-документы для детальных flow-контрактов, статусных моделей,
  схем конфигурации, runtime-поведения и порядка реализации
- если для части знания создан отдельный канонический документ, `README.md`
  должен оставаться кратким summary и ссылаться на этот документ вместо
  дублирования деталей

### 2. Уровень фичи

Каждая значимая фича должна иметь отдельный документ или набор документов.

Для каждой фичи обязателен хотя бы один документ, который отвечает на три
вопроса:

- что строим
- как строим
- как проверяем

Feature-документы должны иметь номер.

Рекомендуемый формат имени файла:

- `docs/features/NNNN-short-title.md`

Если feature-документ остается компактным, достаточно одного файла.

Если feature-документ становится большим, он должен быть преобразован в
директорию с индексным документом и связанными поддокументами.

Рекомендуемый формат:

- `docs/features/NNNN-short-title/README.md`
- `docs/features/NNNN-short-title/*.md`

В таком случае:

- `README.md` выступает входной точкой в feature
- связанные документы выносят детали по отдельным аспектам
- структура должна оставаться привязанной к трем осям:
  `Что строим`, `Как строим`, `Как проверяем`
- `README.md` и соседние документы должны ссылаться друг на друга так, чтобы
  агент или разработчик мог догрузить нужный контекст без ручного поиска по
  репозиторию

Для feature, у которой есть отдельный порядок реализации, рекомендуется
добавлять связанный документ:

- `04-implementation-plan.md`

Требования к нему задаются отдельным документом `docs/implementation-plan.md`.

### 3. Уровень SSOT

Если у фичи есть сложный flow, статусная модель или evolving contract, для нее
должен существовать отдельный SSOT-документ.

Пример:

- `docs/issue-analysis-flow.md`

SSOT может быть сфокусирован на flow или контракте, но он все равно должен быть
привязан к трем главным осям, напрямую или через ссылки на соседние документы.

Если SSOT опирается на соседние feature-документы, ADR или project-level
правила, это должно быть отражено явными ссылками из самого SSOT.

### 4. Уровень ADR

Значимые решения должны оформляться в виде отдельных ADR.

ADR обязателен, если решение меняет или фиксирует:

- архитектуру
- модель данных
- протоколы и интерфейсы
- execution model
- конфигурацию
- интеграцию с внешними системами
- важные ограничения и компромиссы

Требования к ADR:

- каждый ADR имеет номер
- каждый ADR имеет короткий заголовок
- каждый ADR фиксирует статус решения
- ADR должен быть датирован

Рекомендуемый формат имени файла:

- `docs/adr/NNNN-short-title.md`

Уже принятые ADR хранятся в каталоге:

- `docs/adr/`

## Стиль ссылок

Для versioned документации нужно использовать единый стиль внутренних ссылок.

Канонический стиль:

- relative repo-links, например `../README.md` или `./02-how-we-build.md`

Недопустимо:

- использовать absolute filesystem links (например `/home/user/...`)
- использовать web-URL там, где речь идет о versioned документах текущего
  репозитория
