# Governance: порядок приоритетов документов

Статус: active
Роль: SSOT для иерархии документов и правил разрешения конфликтов
Последнее обновление: 2026-03-15

## Назначение

Этот документ фиксирует единый порядок приоритетов документов проекта.

Любой другой документ, упоминающий этот порядок, должен ссылаться сюда, а не
дублировать список.

## Порядок чтения и применения контекста

1. прямой запрос пользователя
2. `AGENTS.md`
3. `AURA.md`
4. `docs/code-quality.md`
5. `README.md`
6. `docs/documentation-structure.md`
7. `docs/documentation-process.md`
8. релевантные SSOT-документы
9. релевантные feature-документы и планы
10. релевантные ADR
11. код, тесты и runtime-артефакты

## Правила применения

- не перепрыгивай к более слабому источнику, если более сильный уже задает
  нужный контракт
- догружай только релевантный слой, а не весь репозиторий подряд
- интерпретируй project-specific документы через призму `AURA.md`, если это не
  разрушает явно зафиксированный контракт проекта
- если между документами разных уровней возникает прямое противоречие, приоритет
  определяется порядком выше; конфликт должен быть указан явно, а не скрыт
  молчаливым компромиссом

## Роли ключевых документов

- `AGENTS.md` — repo-level instruction contract для агентов
- `AURA.md` — личные устойчивые инженерные подходы разработчика
- `docs/code-quality.md` — governance-документ, задающий quality bar сверху вниз;
  код и тесты должны соответствовать ему, а не наоборот
- `README.md` — repo-level входная точка и верхнеуровневый обзор продукта
- `docs/documentation-structure.md` — обязательная структура документации
- `docs/documentation-process.md` — процесс работы с документацией
