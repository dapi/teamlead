# AURA.md

Версия: 0.2
Статус: active
Тип: developer-level guidance
Назначение: project-local точка доступа к личному высокоуровневому видению
разработчика

## Роль

Этот документ задает личные устойчивые подходы разработчика к:

- проектированию информационных систем
- архитектурной декомпозиции
- инженерным компромиссам
- работе с документацией
- оформлению контрактов и решений
- качеству анализа, реализации и проверки

`AURA.md` хранится в репозитории для удобного доступа агента, но не является
описанием истории проекта, продуктового бэклога или локальной runtime-конфигурации.

## Цель

Использовать этот репозиторий как систему с явными слоями знаний, устойчивыми
контрактами и прозрачной инженерной логикой, а не как набор случайно
сложившихся файлов, решений и действий.

## Порядок источников

Читай и используй контекст в таком порядке:

1. прямой запрос пользователя
2. `AGENTS.md`
3. `AURA.md`
4. `README.md`
5. `docs/documentation-structure.md`
6. `docs/documentation-process.md`
7. релевантные SSOT-документы
8. релевантные feature-документы и планы
9. релевантные ADR
10. код, тесты и runtime-артефакты

Правила:

- не перепрыгивай к более слабому источнику, если более сильный уже задает
  нужный контракт
- догружай только релевантный слой, а не весь репозиторий подряд
- интерпретируй project-specific документы через призму `AURA.md`, если это не
  разрушает явно зафиксированный контракт проекта
- если между `AURA.md` и project-specific документом возникает прямое
  противоречие, не скрывай конфликт и укажи его явно

## Базовое направление

В любой системе по умолчанию нужно предпочитать:

- markdown-centric planning and documentation
- contract-first design
- explicit interfaces and invariants
- small, inspectable, testable artifacts
- локальность контекста и предсказуемую навигацию по знаниям
- архитектурные решения, которые можно объяснить, проверить и развивать

Нужно избегать:

- скрытой магии и неявного поведения
- смешивания источников истины
- широких сущностей с несколькими размытыми ответственностями
- случайного дублирования документации
- решений, которые трудно проверить или безопасно изменить

## Дисциплина документации

Документация является частью operating model системы.

Предпочтения:

- канонический источник должен быть один для каждого типа знаний
- верхний уровень должен давать минимально достаточный контекст
- детали должны выноситься в связанные документы по принципу
  `progressive disclosure`
- документы должны разделяться по типам: spec, SSOT, ADR, plan, runbook,
  reference
- лучше ставить ссылку на канонический документ, чем дублировать его текст
- хороший документ должен подсказывать следующее действие

## Документация как Система Измерений

В AI-driven development документацию полезно мыслить не как набор файлов, а как
систему измерений вокруг workflow:

`orient -> specify -> justify -> execute -> verify -> operate`

Смысл этой модели:

- документация должна помогать не только понять систему, но и безопасно менять
  ее
- хороший documentation landscape покрывает и предметную картину, и рабочий
  процесс изменений
- пробелы в документации нужно искать по отсутствующим измерениям workflow, а не
  только по отсутствующим файлам

### Два Слоя

Документация по умолчанию должна раскладываться как минимум на два слоя:

- `project docs`: что это за система, зачем она нужна, какие у нее границы и
  как она устроена
- `flow docs`: как в этой системе работать, как ее менять, как проверять
  результат и когда считать задачу завершенной

Правило:

- без `project docs` агент не понимает предмет, смысл и границы системы
- без `flow docs` агент не понимает процесс работы, критерии остановки и
  expected path изменения системы

### Две Оси

Документы полезно различать по двум ортогональным осям:

- `stable` vs `task-specific`
- `knowledge` vs `behavior-control`

Базовая матрица:

- `stable + knowledge` -> glossary, architecture overview, ADR index
- `stable + behavior-control` -> repo-level instruction contract, module-level
  instruction contract, team skills
- `task-specific + knowledge` -> research notes, profiling results, comparisons
- `task-specific + behavior-control` -> issue, spec, acceptance criteria, exit
  criteria, verification checklist

Правило:

- `knowledge` объясняет, как устроен мир
- `behavior-control` управляет тем, как агент должен действовать
- `stable` должно быть переиспользуемым и малочувствительным к одной задаче
- `task-specific` должно жить рядом с конкретным изменением и его проверкой

### Уровни

Каждый документ должен восприниматься еще и как артефакт определенного уровня:

- `org / program`
- `repo`
- `module / domain`
- `component / service`
- `change / task`

Смешивание нескольких уровней в одном документе допустимо только если это
осознанный компромисс, а не следствие хаотического роста.

### Обязательная Классификация

Перед созданием, чтением, изменением или ссылкой на документ агент по умолчанию
должен быстро классифицировать его по четырем вопросам:

- это `project` или `flow`
- это какой уровень: `org/program`, `repo`, `module/domain`,
  `component/service` или `change/task`
- это `knowledge` или `behavior-control`
- это `stable` или `task-specific`

Эта классификация нужна не для бюрократии, а для инженерных решений:

- где должен жить канонический контракт
- нужен ли новый документ или достаточно обновить существующий
- допустимо ли хранить знание в task-артефакте или его пора поднимать в
  устойчивый слой
- не смешаны ли в одном месте предметное знание и операционные инструкции

### Следствие для Проектирования Документации

Если документация по важной фиче или подсистеме выглядит неполной, сначала
проверь:

- покрыт ли workflow `orient -> specify -> justify -> execute -> verify ->
  operate`
- есть ли и `project docs`, и `flow docs`
- правильно ли разложены знания между `stable` и `task-specific`
- не спрятан ли `behavior-control` внутри документа, который должен быть
  главным образом `knowledge`
- соответствует ли уровень документа реальному масштабу зафиксированного
  контракта

## Contract-First Порядок Работы

Для существенных изменений по умолчанию двигайся в таком порядке:

1. восстановить пользовательское намерение и границы задачи
2. определить, какой документ является каноническим для этого слоя знаний
3. зафиксировать или уточнить контракт в документации
4. определить инварианты, критерии готовности и проверки
5. только после этого менять код
6. затем проверять поведение и выравнивать код с контрактом

Если код и документация расходятся, сначала нужно определить правильный
контракт, а не механически подгонять ответ под текущую реализацию.

## Автономная Feedback-Driven Итерация

Для существенных implementation-задач агент должен мыслить полным циклом, а не
одноразовым patch.

Рабочий цикл:

1. восстановить expectation из сильнейших доступных источников
2. восстановить user intent из текущего запроса и ближайшего task context
3. сформулировать вероятный end-to-end workflow пользователя
4. выполнить ближайший безопасный шаг
5. собрать feedback из тестов, логов, поведения системы, диагностики и артефактов
6. сравнить actual result с expectation и user intent
7. выявить оставшиеся gaps
8. повторить цикл, если следующий шаг ясен и безопасен

Правила:

- feedback должен быть измерительным слоем между ожидаемым и фактическим
  результатом
- если следующий шаг ясен и безопасен, не останавливайся на полпути
- если безопасное продолжение блокируется, явно показывай blocker
- нельзя выдумывать требования, если есть более сильный пользовательский или
  repo-level контекст

## Дисциплина Завершения Сессии

Предпочтительный стиль завершения implementation session:

- держать изменения атомарными
- не захватывать в commit нерелевантные правки
- перед commit упрощать решение без изменения поведения, если это снижает
  сложность
- перед завершением проверять затронутые контракты, код и документацию
- не оставлять незавершенную работу в виде замаскированного "готово", если
  остаются существенные gaps

## Границы Документа

`AURA.md` не фиксирует:

- текущие статусы задач
- project-specific runtime-состояние
- конкретные пути, команды и конфигурационные значения проекта
- продуктовые договоренности конкретной фичи
- локальные исключения, уже зафиксированные в SSOT или ADR

Эти знания должны жить в project-specific документах репозитория.
