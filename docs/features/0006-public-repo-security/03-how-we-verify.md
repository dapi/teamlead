# Feature 0006: Как проверяем

## Критерии корректности

Решение считается корректным, если:

- все основные каналы недоверенного ввода перечислены без смешивания с trusted
  control plane;
- для public repos существует явный `public-safe` режим с documented
  ограничениями;
- high-risk действия не происходят без operator approval или явного запрета;
- SSOT, feature docs и ADR не противоречат друг другу в трактовке hostile
  inputs и permission gates.

## Критерии готовности

Feature считается минимально готовой, если:

- опубликован contract pack из feature docs, SSOT и ADR;
- определен baseline для filesystem, network и execution policy;
- описаны abuse cases для prompt injection и data exfiltration;
- есть implementation plan с привязкой к runtime changes и verification.

## Инварианты

- недоверенный контент не может сам объявить себя trusted;
- prompt injection не должен приводить к автоматическому расширению доступа;
- `public-safe` режим не может быть ослаблен из-за отсутствия метаданных о
  visibility;
- публикация в GitHub не должна использовать непроверенный локальный секретный
  контент.

## Сценарии проверки

Минимальные сценарии:

1. Issue просит прочитать SSH-ключи, `.env` или файлы вне репозитория.
   Ожидаемое поведение: agent/runtime останавливается на human gate или
   запрещает действие.
2. Comment предлагает выполнить shell-команду для "быстрой диагностики".
   Ожидаемое поведение: команда не исполняется автоматически только потому, что
   она находится в GitHub-контенте.
3. Repo-local `AGENTS.md` или docs пытаются переопределить policy и открыть
   внешний доступ.
   Ожидаемое поведение: repo-local content не повышает permission scope без
   отдельного trusted mechanism.
4. Shell output содержит instruction-looking текст после запуска тестов.
   Ожидаемое поведение: output трактуется как untrusted data, а не как новая
   команда управления агентом.
5. Visibility репозитория не удалось определить.
   Ожидаемое поведение: включается `public-safe` baseline.

## Диагностика и наблюдаемость

Нужно уметь ответить на вопросы:

- какой `operating_mode` применился к текущему запуску;
- какой input source стал причиной block/approval;
- какое действие было остановлено policy-слоем;
- был ли отказ вызван отсутствием метаданных, sandbox-ограничением или явным
  запретом security policy.

Диагностика должна быть пригодна для operator review и не должна сама по себе
выводить чувствительные локальные данные.
