# ADR-0006: Использовать `gh` CLI как слой интеграции с GitHub для MVP

Статус: accepted
Дата: 2026-03-13

## Контекст

Для первой версии `ai-teamlead` нужно определить, через какой механизм
инструмент работает с GitHub issues и GitHub Project.

Рассматривались варианты:

- прямой вызов GitHub REST/GraphQL API через собственный клиент
- использование `gh` CLI как основного интеграционного слоя
- гибридный подход с несколькими backend-реализациями с самого начала

Требования:

- решение должно быть простым для MVP
- авторизация должна быть уже решаема существующей пользовательской средой
- отладка должна быть удобной из терминала
- решение должно хорошо подходить для personal Linux-first workflow

## Решение

В MVP `ai-teamlead` использует `gh` CLI как основной слой интеграции с GitHub.

Через `gh` должны решаться как минимум:

- чтение информации о текущем репозитории
- поиск и чтение issues
- чтение и изменение статусов GitHub Project
- публикация изменений состояния, если это потребуется flow

Прямой собственный GitHub API client не вводится в первой версии.

## Последствия

Плюсы:

- меньше кода и меньше проблем с авторизацией
- проще ручная отладка и диагностика
- решение естественно вписывается в terminal-first workflow
- проще использовать уже существующую авторизацию пользователя в `gh`

Минусы:

- появляется жесткая зависимость от установленного `gh`
- формат интеграции завязан на shell execution
- в будущем может понадобиться отдельный native API layer
