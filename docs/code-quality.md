# Критерии качества кода

## Назначение

Этот документ задает минимальный quality bar для кода в проекте.

Цель:

- сделать требования к качеству явными
- заранее определить, какие тесты обязательны
- отделить MVP-допустимые компромиссы от недопустимой халтуры

## Общие критерии качества

Код считается приемлемым, если он:

- читаем и разбит на понятные модули
- не скрывает ключевые побочные эффекты
- не смешивает бизнес-логику, shell execution и парсинг вывода в одном месте
- дает детерминированное поведение при одинаковом входе
- явно обрабатывает ошибки внешних зависимостей
- не использует неявный глобальный state
- следует зафиксированным ADR, SSOT и feature-спекам

## Что обязательно для архитектуры

Нужно стремиться к такой структуре:

- чистая логика отдельно
- адаптеры к `gh`, `zellij`, файловой системе и git отдельно
- entrypoint и orchestration отдельно
- формат runtime-артефактов отдельно от логики flow

Недопустимо:

- размазывать shell-команды по всему коду без единого abstraction layer
- привязывать логику к текущему process state, если по контракту источник истины
  находится вне процесса
- шить repo-specific значения в код

## Требования к тестам

Для каждой значимой части кода нужно выбрать подходящий уровень тестов:

1. Unit tests
2. Integration tests
3. Smoke tests

Не каждая функция требует всех трех уровней, но каждая важная возможность
должна быть покрыта достаточной комбинацией.

Для каждой `feature`, каждого `bugfix` и каждого значимого изменения
обязателен хотя бы один тест:

- либо воспроизводящий исправляемую проблему
- либо проверяющий целевой `use case`, ради которого делается изменение

Изменение без такого теста не соответствует quality bar проекта.

## Обязательные unit tests для MVP

Обязательны тесты для:

- загрузки и валидации `./.ai-teamlead/settings.yml`
- определения repo context из текущего git-репозитория
- выбора issue из `Backlog` в детерминированном порядке snapshot GitHub Project
- проверки допустимости переходов для `run`
- чтения и записи runtime/session-артефактов
- проверки invariant `issue <-> session_uuid`
- интерпретации operator events для waiting-статусов

## Обязательные integration tests для MVP

Обязательны тесты для:

- работы CLI `poll` на тестовом runtime-контексте
- работы CLI `run` на тестовом runtime-контексте
- взаимодействия с оберткой над `gh`
- создания runtime-артефактов в `.git/.ai-teamlead/`
- запуска launcher-контракта для `zellij` без требования real end-to-end UI

Для MVP допустимо использовать test doubles или fake adapters там, где прямой
e2e с `gh` или `zellij` слишком дорог в поддержке.

## Обязательные smoke tests для MVP

Перед приемкой MVP нужно пройти вручную минимум:

- запуск `ai-teamlead poll` в реальном репозитории
- успешный `poll` для issue в `Backlog`
- успешный `run` для допустимой issue
- отказ `run` для недопустимой issue
- проверку создания session-binding и runtime-артефактов
- проверку, что два репозитория могут работать независимо

## Что можно временно не покрывать

Для MVP допустимо не требовать:

- полный e2e поверх реального GitHub Project API на каждый коммит
- полный e2e поверх реального `zellij`
- тесты на редкие UX-сообщения, если покрыта сама логика ветвления

Но нельзя пропускать тесты для ключевых переходов состояний и session-binding.

## Definition Of Done для кода

Изменение считается готовым, если:

- код соответствует текущим документированным контрактам
- есть тесты нужного уровня для новой или измененной логики
- для каждой `feature`, каждого `bugfix` и каждого значимого изменения есть
  тест, который либо воспроизводит проблему, либо проверяет целевой `use case`
- не нарушены существующие инварианты
- диагностические сообщения достаточны для ручной отладки
- change set не вносит неописанную архитектурную магию

## Feature-first правило

Если добавляется новая feature или меняется важный flow, нужно обновить:

- feature-спеку
- SSOT при необходимости
- ADR при необходимости
- тесты

Код без синхронизации с документацией считается неполным.
