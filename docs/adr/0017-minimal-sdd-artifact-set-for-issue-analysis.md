# ADR-0017: минимальный набор SDD-артефактов для `issue-analysis-flow`

Статус: accepted
Дата: 2026-03-13

## Контекст

`issue-analysis-flow` должен создавать не просто произвольный план в чате, а
versioned analysis-артефакты, достаточные для дальнейшей автоматической
разработки issue агентами.

При этом первая версия должна оставаться компактной. Нельзя требовать
чрезмерный пакет документов для каждой issue, но и нельзя оставлять результат
анализа только в виде свободного текста без структуры.

В проекте уже принят базовый documentation framework по трем осям:

- что строим
- как строим
- как проверяем

## Решение

Для каждого analysis run создается минимальный versioned SDD-комплект в
каталоге:

- `specs/issues/${ISSUE_NUMBER}/`

Обязательный минимум:

- `README.md`
- `01-what-we-build.md`
- `02-how-we-build.md`
- `03-how-we-verify.md`

Правило минимальности:

- на каждую из трех осей должен существовать минимум один документ
- по умолчанию это именно три базовых документа выше
- дополнительные документы добавляются только если issue этого действительно
  требует

## Семантика документов

`01-what-we-build.md`:

- проблема
- пользователь или роль
- scope и non-goals
- user story / use cases / BDD элементы в минимально достаточном объеме

`02-how-we-build.md`:

- архитектурный подход
- данные и интерфейсы
- domain model и технические решения в минимально достаточном объеме
- ссылки на ADR, если по issue появляются новые решения

`03-how-we-verify.md`:

- acceptance criteria
- test plan
- QA checklist или equivalent verification strategy

`README.md`:

- краткое резюме issue
- ссылка на GitHub issue
- перечисление артефактов по трем осям
- статус анализа

## Последствия

Плюсы:

- analysis output становится versioned и машиночитаемо-структурированным на
  уровне layout
- сохраняется совместимость с SDD-подходом
- результат анализа становится пригодным для будущего implementation flow

Минусы:

- даже для маленьких issue появляется минимальный набор файлов
- launcher и prompt должны явно направлять агента на создание этих документов

## Альтернативы

### 1. Один общий `analysis.md`

Отклонено.

Это размывает три оси и ухудшает применимость артефактов для автоматической
разработки.

### 2. Большой фиксированный пакет документов на каждую issue

Отклонено.

Для первой версии это избыточно и будет порождать формальный шум.
