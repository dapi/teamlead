## Problem

Сейчас bootstrapped `./.ai-teamlead/settings.yml` создается как полностью
активный YAML с рабочими значениями почти для всех полей. Это приводит к трем
проблемам:

- стартовый шаблон одновременно считается runtime-конфигом и документацией по
  доступным настройкам;
- default-значения фактически размазаны между Rust-кодом, шаблоном
  `templates/init/settings.yml` и repo-level документацией;
- при добавлении нового key разработчик может обновить только один слой и не
  заметить расхождение между реальным поведением приложения и bootstrap
  шаблоном.

В результате `init` не дает настоящий zero-config для полей, у которых у
приложения уже есть естественные встроенные defaults, а эволюция схемы
становится хрупкой и плохо проверяемой.

## Who Is It For

Фича нужна двум ролям:

- владельцу или оператору репозитория, который запускает `ai-teamlead init` и
  хочет получить понятный versioned обзор настроек без необходимости
  разкомментировать каждое defaulted поле;
- разработчику `ai-teamlead`, который развивает схему `settings.yml` и должен
  иметь один явный контракт: какие поля обязательны, какие defaulted, а какие
  присутствуют в шаблоне только как opt-in пример.

## User Story

Как владелец репозитория, использующий `ai-teamlead`, я хочу получить после
`init` понятный `settings.yml`, где обязательные поля и возможные настройки
видны как документированный шаблон, а runtime автоматически подставляет
канонические defaults для defaulted-полей, чтобы стартовая настройка не
требовала ручного разкомментирования значений, которые и так известны
приложению.

## Outcome

Целевой результат:

- `./.ai-teamlead/settings.yml` можно оставить почти или полностью в
  закомментированном виде для всех defaulted-полей;
- runtime загружает missing defaulted-поля из одного canonical default-layer;
- обязательные repo-specific поля имеют отдельный явный контракт и понятную
  ошибку, если пользователь их не задал;
- template, код и документация не расходятся тихо, потому что guardrail ломает
  сборку при неполном обновлении схемы.

## Use Cases

1. Пользователь выполняет `ai-teamlead init` в новом репозитории и получает
   `settings.yml`, где большинство настроек показаны как комментарии, а
   runtime использует встроенные defaults без ручного редактирования.
2. Пользователь задает только `github.project_id`, оставляя остальные поля
   неактивными, и приложение продолжает работать на canonical defaults.
3. Разработчик добавляет новый config key и обязан явно решить, является ли он
   required-полем, runtime default или только example-only extension в шаблоне.
4. Старый репозиторий с уже материализованным полным `settings.yml`
   продолжает десериализоваться без миграции и без смены поведения.

## Scope

В первую версию входят:

- явная классификация schema keys на `required-without-default`,
  `defaulted-by-application` и допустимые `example-only extension`;
- zero-config контракт для defaulted-полей, включая возможность оставить
  стартовый шаблон без активных значений для этих полей;
- canonical default-layer приложения, из которого runtime берет отсутствующие
  значения;
- обновление шаблона `templates/init/settings.yml`, чтобы он отражал доступные
  настройки и текущие defaults в закомментированном виде;
- guardrail-тесты на консистентность схемы, default-layer и шаблона;
- явное описание required-полей и диагностик для них в документации.

## Non-Goals

В эту задачу не входят:

- отказ от `./.ai-teamlead/settings.yml` как versioned repo-local конфига;
- интерактивный мастер `init` для заполнения обязательных значений;
- перенос repo context, owner/repo или других target-параметров в глобальный
  пользовательский конфиг;
- автоматическая миграция уже существующих репозиториев с кастомным
  `settings.yml`;
- универсальный генератор комментариев или внешняя schema registry вне
  текущего Rust-приложения;
- добавление новых продуктовых настроек, не нужных для решения drift-проблемы.

## Constraints And Assumptions

- Источник истины для runtime defaults должен быть один и находиться внутри
  приложения, а не в bootstrap YAML.
- Repo-specific поля без естественного default, в первую очередь
  `github.project_id`, не должны получать скрытый fallback.
- `settings.yml` остается versioned документом project-local contract layer и
  по-прежнему создается командой `ai-teamlead init`.
- Старые fully materialized конфиги должны оставаться обратно совместимыми.
- Если в шаблоне появляется комментарий, который не совпадает с фактическим
  runtime default, это допустимо только для явно помеченного opt-in примера,
  который не меняет поведение по умолчанию.
