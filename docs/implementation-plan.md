# Требования к документу «План имплементации»

## Назначение

Этот документ задает единые требования к артефакту `План имплементации`.

`План имплементации` нужен как мост между документацией уровня
`Что строим` / `Как строим` / `Как проверяем` и фактическим порядком
реализации.

Он должен помогать агенту или разработчику быстро понять:

- что именно нужно делать
- в каком порядке это делать
- на какие документы и решения опирается каждый шаг
- как прослеживается причинно-следственная связь между решением, кодом и
  проверкой

## Когда документ обязателен

`План имплементации` обязателен, если:

- задача дошла до стадии планирования реализации
- по issue формируется план, который должен пройти human review
- для значимой feature или подсистемы нужна явная последовательность этапов
  реализации
- изменение затрагивает несколько документов, подсистем или решений и без
  отдельного плана теряется прослеживаемость

Для мелких локальных изменений допустим компактный план, но требования к
прослеживаемости и ссылкам сохраняются.

## Роль документа

`План имплементации`:

- не заменяет feature-спеку, SSOT или ADR
- не дублирует их целиком
- использует `progressive disclosure`: кратко формулирует шаг и ссылается на
  источник решения или требования
- связывает реализацию с тестами, критериями готовности и ограничениями

Если для шага нет документа, объясняющего почему он нужен, это сигнал либо
добавить ссылку на существующий документ, либо сначала создать недостающую
документацию.

## Обязательные требования

В каждом `Плане имплементации` должны быть:

- явное назначение плана и его scope
- ссылки на все документы, без которых нельзя корректно реализовать задачу
- явный план изменений документации или зафиксированное обоснование, почему
  документация не меняется
- порядок этапов или шагов реализации
- для каждого этапа: цель, ожидаемый результат и основание в документации
- описание проверки и тестов для каждого значимого изменения
- критерий завершения
- открытые вопросы, риски или зависимости, если они влияют на порядок работ

План считается неполным, если по нему нельзя быстро восстановить:

- почему этап существует
- каким документом он обоснован
- какой тест или сценарий подтверждает результат этапа
- какие документы должны быть обновлены вместе с кодом или почему этого не
  требуется

## План изменений документации

В каждом `Плане имплементации` должен быть отдельный раздел или явный блок,
который отвечает на вопрос, как изменение затронет documentation landscape.

Минимум в этом разделе нужно зафиксировать:

- какие канонические документы нужно обновить до кода или вместе с кодом:
  SSOT, feature-docs, ADR, verification-документы, `README.md`
- какие производные summary-документы, шаблоны или repo-local assets нужно
  синхронизировать после обновления канонического источника
- какие документы сознательно не меняются и почему это допустимо для текущей
  задачи

Раздел не должен превращаться в дублирование всего плана. Его задача:

- сделать documentation delta явным до начала реализации
- не позволить забыть update канонического контракта и его производных слоев
- показать, где изменение требует новой документации, а где достаточно
  синхронизации уже существующих summary

## Обязательные ссылки

`План имплементации` должен содержать ссылки на все релевантные документы из
следующего набора:

- GitHub issue или локальный issue-документ
- feature-спека или issue-спека
- SSOT-документы по flow, state machine или контрактам
- ADR, которые объясняют ключевые решения и ограничения
- документы `Как проверяем` и `docs/code-quality.md`
- связанные implementation plans или dependency features, если текущий план
  зависит от них

Если документ не нужен для понимания причинно-следственных связей задачи, его
не нужно добавлять ради объема.

## Требование к причинно-следственной прослеживаемости

Каждый значимый этап или группа шагов должны отвечать на три вопроса:

1. Почему это нужно.
2. На какой документ или решение это опирается.
3. Как будет проверено, что этап реализован корректно.

Для этого у этапа должны быть явно указаны:

- `Цель`
- `Результат этапа`
- `Основание`
- при необходимости `Проверка` или ссылка на тестовые сценарии

Если агент открывает план отдельно от остальной документации, он должен по
ссылкам быстро находить первоисточник решения без ручного поиска по
репозиторию.

## Рекомендуемая структура

Минимальная структура документа:

- заголовок
- статус и дата последнего обновления
- назначение
- scope
- связанные документы
- план изменений документации
- зависимости и предпосылки
- порядок работ
- критерий завершения
- открытые вопросы или риски
- журнал изменений

Для планов с несколькими этапами каждый этап должен содержать минимум:

- `Цель`
- `Основание`
- `Результат этапа`

Если этап существенно меняет поведение, интерфейсы или runtime, нужно также
добавлять:

- `Проверка`
- ссылки на тесты, smoke-сценарии или verification-документы

## Правила качества

Хороший `План имплементации`:

- не перегружает читателя деталями раньше времени
- раскрывает детали по мере углубления в этапы
- не смешивает продуктовый контракт с порядком работ
- не оставляет шаги без обоснования
- не содержит этапов, происхождение которых нельзя объяснить ссылкой на
  документацию

Плохой `План имплементации`:

- перечисляет шаги без объяснения причин
- ссылается только на себя
- заставляет вручную искать связанные ADR и SSOT по всему репозиторию
- не связывает изменения с тестами и критериями готовности
