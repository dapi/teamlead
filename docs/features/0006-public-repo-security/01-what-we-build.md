# Feature 0006: Что строим

## Проблема

`ai-teamlead` читает недоверенный контент из GitHub и репозитория, а затем
работает в локальном окружении пользователя.

Для публичных репозиториев это означает, что злоумышленник может:

- вставить prompt injection в issue, комментарии, linked artifacts или файлы
  репозитория;
- попытаться заставить агента выполнить нежелательные shell-команды;
- спровоцировать чтение чувствительных файлов и утечку их содержимого;
- навязать агенту ложный analysis или implementation path.

Сейчас сама возможность использования `ai-teamlead` на public repos не должна
подразумевать, что любой входной текст безопасен или что агент может без
ограничений доверять repo-local документам.

## Пользователь

Основные пользователи:

- владелец `ai-teamlead`, который хочет запускать workflow на внешних
  репозиториях;
- оператор, который вручную запускает `run` или `poll`;
- разработчик, использующий локальный агентский workflow внутри подключенного
  репозитория.

## Результат

Полезным результатом считается security baseline, в котором:

- все основные каналы недоверенного ввода перечислены и классифицированы;
- существует явный safe operating mode для public repos;
- недоверенный контент не может напрямую менять tool policy или permission
  model;
- high-risk действия проходят через human gate или запрещаются;
- архитектурные ограничения зафиксированы в SSOT и ADR, а не остаются
  неявными допущениями.

## Scope

В первую версию входит:

- threat model для public-repo сценария;
- классификация trusted и untrusted inputs;
- safe mode baseline для локального запуска;
- правила human approval для опасных действий;
- ограничения на filesystem scope, network access и external link handling;
- минимальные требования к verification для hostile inputs;
- выделение связанных feature docs, SSOT и ADR.

## Вне scope

- полная формальная модель угроз для всех поддерживаемых LLM-провайдеров;
- универсальная sandbox-платформа вне текущей execution model проекта;
- автоматическая проверка или очистка всего недоверенного контента;
- защита от скомпрометированной локальной машины пользователя;
- гарантии безопасности для произвольных сторонних tools вне контракта
  `ai-teamlead`.

## Ограничения и предпосылки

- public repo должен рассматриваться как hostile-by-default источник контента;
- issue, comments, linked artifacts и repo-local docs не являются trust anchor
  сами по себе;
- оператор может сознательно разрешать отдельные действия, но это не должно
  происходить неявно;
- SSOT и ADR должны появиться раньше или одновременно с runtime-изменениями;
- до появления подтвержденного safe mode поддержка public repos не считается
  production-ready.
