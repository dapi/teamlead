# Процесс работы с документацией

## Назначение

Этот документ описывает как и когда работать с документацией проекта.

Структура документации (что и где должно быть) описана в
`docs/documentation-structure.md`.

## Правила для фич

Для каждой новой значимой фичи нужно:

1. создать отдельный feature-документ
2. описать в нем три оси
3. если нужно, выделить отдельный SSOT для flow или state machine
4. зафиксировать критерии проверки до начала реализации
5. оформить отдельные значимые решения как ADR
6. если документ разрастается, вынести его в отдельную feature-директорию

Если фича развивается существенно, ее документация должна обновляться раньше
или одновременно с кодом, но не задним числом постфактум.

Если задача дошла до стадии планирования реализации, должен быть создан
отдельный `План имплементации`, оформленный по правилам
`docs/implementation-plan.md`.

## Принцип progressive disclosure

Документация создается по принципу `progressive disclosure`:

- входной документ должен давать минимально достаточную картину без перегрузки
  деталями
- детали должны раскрываться в отдельных связанных документах по мере
  необходимости
- более глубокие уровни документации не должны дублировать краткий контекст из
  входной точки, а должны уточнять и расширять его

Если для правильного понимания текущего документа агенту или разработчику нужен
дополнительный контекст, в документе должны стоять явные ссылки на связанные
SSOT, feature-документы, ADR или соседние разделы.

Это правило отдельно обязательно для `Плана имплементации`: при создании задачи
и при планировании ее исполнения в плане нужно ставить ссылки на все документы,
нужные для быстрого восстановления причинно-следственных связей решений.

## Статус README

`README.md` является частью project docs и repo-level входной точкой, но не
универсальным SSOT для всех слоев знаний.

Правила:

- `README.md` является каноническим источником только для верхнеуровневого
  описания продукта, границ MVP, общей карты документации и краткой
  архитектурной рамки репозитория
- если меняется repo-level позиционирование проекта, границы MVP или состав
  базовых документов, сначала или одновременно обновляется `README.md`
- если меняется flow, state machine, статусная модель, схема конфигурации,
  subsystem-level контракт, runtime-поведение или порядок реализации, сначала
  обновляется профильный канонический документ: SSOT, feature-документ, ADR или
  `План имплементации`
- после обновления профильного канонического документа `README.md`
  синхронизируется только как краткая summary и набор ссылок на первоисточники,
  если это изменение важно для repo-level ориентирования
- если между `README.md` и более узким каноническим документом возникает
  расхождение, источником истины считается профильный документ в своей области
  ответственности, а `README.md` должен быть выровнен с ним
- при эволюции документации `README.md` не должен разрастаться в дублирующий
  реестр статусов, конфигов, переходов и implementation-деталей

## Шаблон feature-документа

Каркас для новых фич: `docs/templates/feature-spec-template.md`.

Каркас для планов реализации: `docs/templates/implementation-plan-template.md`.

## Связь с реализацией

Документация должна управлять реализацией, а не объяснять уже случайно
сложившийся код.

Если код и документация расходятся:

- сначала нужно определить, что считается правильным контрактом
- затем выровнять либо код, либо документацию

## Работа с ADR

ADR фиксирует одно конкретное решение в конкретном контексте. ADR — это
самодостаточный документ, а не реестр функциональности.

Продукт может развиваться после принятия ADR: добавляться новые команды,
аргументы, опции. Это не является поводом обновлять или расширять существующий
ADR. Новый ADR создается только если принимается новое значимое решение.
