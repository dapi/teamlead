# doc-review-team

Параллельное ревью документации через Agent Team.

## Когда использовать

- Пользователь просит проверить документацию на gaps, полноту, противоречия, консистентность
- Пользователь просит сводный отчёт по качеству документации
- Пользователь вызывает `/review-docs`

## Workflow

### Шаг 1. Определить targets

Если пользователь указал конкретные пути — использовать их.
Иначе — дефолтные targets:

- `README.md`
- `docs/`

### Шаг 2. Создать team

```
TeamCreate: team_name="doc-review", description="Параллельное ревью документации"
```

### Шаг 3. Создать задачи

Создать 5 задач через TaskCreate:

1. **gaps** — поиск смысловых и структурных пробелов
2. **completeness** — проверка полноты относительно governance-правил
3. **contradictions** — поиск противоречий между документами
4. **consistency** — проверка консистентности терминологии и формы
5. **verification** — верификация findings (blocked by 1-4)

### Шаг 4. Запустить 4 агентов ПАРАЛЛЕЛЬНО

Запустить 4 Agent-вызова в ОДНОМ сообщении (параллельно). Каждый агент:

- `subagent_type`: `general-purpose`
- `team_name`: `doc-review`
- `run_in_background`: `true`
- `name`: `reviewer-{aspect}`
- `mode`: `plan` (агент не может редактировать файлы без одобрения)

Каждому агенту передать промпт из раздела «Промпты аспектов» ниже, подставив актуальные targets.

### Шаг 5. Дождаться завершения

Все 4 агента запущены в background. Дождаться уведомлений о завершении каждого.

### Шаг 6. Предварительный синтез

После получения всех 4 результатов — собрать черновой список:

1. Отсечь findings с confidence: low — они не попадают в итоговый отчёт
2. Дедуплицировать одинаковые findings из разных аспектов
3. Объединить findings, ссылающиеся на одну проблему
4. Составить список кандидатов для верификации (severity high или medium)

### Шаг 7. Верификация findings

Запустить одного verification-агента (в background):

- `subagent_type`: `general-purpose`
- `team_name`: `doc-review`
- `run_in_background`: `true`
- `name`: `verifier`
- `mode`: `plan`

Промпт верификатора:

```
Ты верификатор findings из документационного ревью.
Для каждого finding ниже:
1. Открой указанные файлы и проверь evidence — действительно ли проблема существует.
2. Поставь verdict: confirmed, downgraded или false-positive.
   - confirmed — проблема подтверждена, evidence корректен
   - downgraded — проблема есть, но severity завышена (укажи новую severity)
   - false-positive — проблема не подтверждена, evidence некорректен или устарел
3. Для каждого verdict дай краткое обоснование (1-2 предложения).

Отвечай на русском. Финальный ответ — не более 2000 символов.
Отправь результат team lead-у через SendMessage (type: "message", recipient: "team-lead").
Отметь свою задачу как completed через TaskUpdate.
```

К промпту приложить список findings (severity high/medium) из шага 6.

### Шаг 8. Итоговый отчёт

После получения результатов верификации — собрать финальный отчёт:

1. Исключить false-positive findings
2. Применить downgrade если верификатор понизил severity
3. Оставить confirmed findings как есть

Формат итогового отчёта:

```
# Documentation Review Summary

## Критичные findings (high)
- ...

## Средние findings (medium)
- ...

## Низкие findings (low)
- ...

## Отсеянные (false-positive)
- ... (кратко, для прозрачности)

## Что чинить сначала
1. ...
2. ...
3. ...
```

### Шаг 9. Завершить team

Отправить shutdown_request всем тиммейтам, затем TeamDelete.

## Промпты аспектов

Каждому агенту передаётся промпт, состоящий из:
1. Общая преамбула
2. Фокус аспекта
3. Список governance-документов
4. Список targets
5. Требования к ответу

### Общая преамбула

```
Ты делаешь reviewer-only аудит документации репозитория.
Ничего не меняй; нужен только список findings.
Отвечай на русском языке.

ВАЖНО: Когда закончишь анализ, ОБЯЗАТЕЛЬНО отправь свои findings team lead-у
через SendMessage (type: "message", recipient: "team-lead").
Также отметь свою задачу как completed через TaskUpdate.
```

### Фокус по аспектам

**gaps:**
```
Фокус: gaps (пробелы)
Ищи смысловые и структурные пробелы: отсутствующие разделы, недоописанные контракты, пропущенные зависимости, неописанные ограничения и missing links между документами.
```

**completeness:**
```
Фокус: completeness (полнота)
Проверяй полноту относительно repo-level правил. Смотри, покрыты ли оси «Что строим», «Как строим», «Как проверяем», хватает ли критериев готовности, инвариантов, сценариев проверки и явного scope.
```

**contradictions:**
```
Фокус: contradictions (противоречия)
Ищи противоречия между документами: несовместимые статусы, разные naming contracts, конфликтующие требования, различающиеся promises about behavior, conflicting source-of-truth claims.
```

**consistency:**
```
Фокус: consistency (консистентность)
Проверяй консистентность формы и терминологии: одинаковые имена статусов, единый стиль ссылок, согласованность терминов feature/flow/ADR, одинаковое использование путей, конфиг-ключей и runtime-понятий.
```

### Governance-документы

Агент должен сначала прочитать (если существуют):

- `AGENTS.md`
- `README.md`
- `docs/documentation-structure.md`
- `docs/code-quality.md`
- `docs/issue-analysis-flow.md`

### Требования к ответу агента

```
Требования к ответу:
1. Дай только findings по делу, без длинного пересказа.
2. У каждого finding должны быть:
   - Severity: high, medium или low
   - Confidence: high, medium или low (насколько ты уверен, что это реальная проблема)
   - Короткий заголовок
   - Почему это проблема
   - Evidence с точными file references (файл, строка или секция)
   - Что нужно уточнить или дописать
3. Confidence guide:
   - high — прямое противоречие или явный пропуск, подтверждённый цитатой из документа
   - medium — вероятная проблема, но требует интерпретации или контекста
   - low — возможная проблема, основанная на предположении или неявном ожидании
4. Если явных проблем нет, напиши: "Findings: none".
5. Не дублируй одно и то же наблюдение в нескольких формулировках.
6. Финальный ответ — не более 2000 символов. Список outcomes, не process.
```

## Правила

- Это reviewer-only workflow: НЕ менять документацию без отдельного запроса пользователя.
- Если пользователь просит проверить только часть docs — передавать только указанные пути.
- Агенты запускаются как `general-purpose` в режиме `plan` — они имеют доступ к SendMessage и TaskUpdate для коммуникации, но не могут редактировать файлы без одобрения.
