# ADR-0013: История агентской сессии как источник диалога

Статус: accepted
Дата: 2026-03-13

## Контекст

Ранее в runtime-модели появились отдельные durable-артефакты:

- `questions.json`
- `analysis-plan.json`
- `operator-events.jsonl`

Они предполагались как machine-readable представление вопросов, плана и действий
оператора.

Но в принятой пользовательской модели `issue-analysis-flow` агент работает в
интерактивной агентской сессии, привязанной к конкретной issue. Вопросы,
ответы, уточнения и обсуждение плана происходят прямо внутри этой сессии.

## Решение

Для MVP принимается следующая модель:

- источником истины по диалогу является история агентской сессии
- источником истины по статусу issue является GitHub Project status
- локальный runtime state хранит только session-binding и технические
  артефакты запуска

Из обязательной runtime-модели MVP исключаются:

- `questions.json`
- `analysis-plan.json`
- `operator-events.jsonl`

Команда `run` в waiting-статусах трактуется как явный операторский сигнал
продолжить или переоткрыть анализ в уже существующей issue-сессии.

## Последствия

Плюсы:

- модель MVP становится проще и ближе к реальному UX
- не нужно дублировать содержимое диалога в отдельных JSON-файлах
- снижается объем runtime state и число синхронизационных проблем

Минусы:

- automation не получает machine-readable представление вопросов и плана
- сложнее строить автоматические проверки поверх human gate
- позднее, возможно, придется вернуть структурированные артефакты как
  дополнительный слой

## Альтернативы

### 1. Оставить JSON-артефакты обязательными уже в MVP

Отклонено.

Это усложняет систему без доказанной пользы для текущего интерактивного flow.

### 2. Сделать историю агентской сессии вторичным источником, а JSON-файлы основным

Отклонено.

Это противоречит целевому интерактивному UX и ведет к двойному источнику
истины по содержимому анализа.
