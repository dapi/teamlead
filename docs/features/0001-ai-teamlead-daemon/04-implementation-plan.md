# Feature 0001: План реализации

Статус: draft
Последнее обновление: 2026-03-13

## Назначение

Этот документ задает порядок реализации MVP для `ai-teamlead daemon`.

В отличие от `02-how-we-build.md`, этот документ описывает не целевую
архитектуру, а последовательность практических шагов реализации.

## Принцип порядка работ

Реализация должна идти от более стабильных и низкорисковых частей к более
интеграционным:

1. config и repo context
2. GitHub integration
3. базовый polling cycle
4. ручные команды
5. запуск flow через `zellij`
6. диагностика и минимальная полировка

## Этап 1. Каркас daemon и загрузка конфига

Цель:

- иметь запускаемый process entrypoint
- уметь находить текущий репозиторий
- читать `ai-teamlead.yml`
- валидировать минимальный конфиг

Результат этапа:

- есть entrypoint daemon
- есть config loader
- есть проверка, что запуск идет в git-репозитории
- есть резолв repo-local paths

## Этап 2. Определение repo context

Цель:

- корректно определять текущий git-репозиторий
- получать owner/repo по умолчанию из git remote

Результат этапа:

- есть стабильный repo context object
- обработаны ошибки отсутствующего git remote
- документировано, что считается текущим репозиторием
- GitHub target жестко совпадает с текущим git-репозиторием

## Этап 3. Интеграция с `gh`

Цель:

- проверить доступность `gh`
- читать metadata по репозиторию
- уметь искать issues и читать их состояние
- уметь читать и менять статус в GitHub Project

Результат этапа:

- есть GitHub adapter на базе `gh`
- есть базовая обработка ошибок shell execution и auth
- подтвержден рабочий path для чтения и смены статусов

## Этап 4. Базовый `poll`

Цель:

- реализовать один polling cycle
- выбирать только подходящую issue из `Backlog`
- переводить issue в `Analysis In Progress`

Результат этапа:

- работает ручная команда `poll`
- при отсутствии задач команда корректно завершается без ошибок
- при ошибке claim flow не стартует

## Этап 5. Базовый `run`

Цель:

- реализовать ручной запуск по issue number или issue URL
- соблюдать допустимые входные статусы
- не обходить transition rules

Результат этапа:

- работает ручная команда `run`
- недопустимые запуски отклоняются с понятной диагностикой

## Этап 6. Launcher для `issue-analysis-flow`

Цель:

- запускать flow в настроенной `zellij` session и tab
- передавать issue context в agent session
- генерировать и сохранять `session_uuid`
- создавать durable-связку `issue <-> session_uuid`
- сохранять `zellij.session_id`, `zellij.tab_id`, `zellij.pane_id`

Результат этапа:

- после успешного claim запускается agent session
- ошибки интеграции с `zellij` видны оператору
- вопросы, план и операторские действия привязываются к session-артефактам

## Этап 7. Основной daemon loop

Цель:

- обернуть `poll` в foreground loop
- реализовать `runtime.poll_interval_seconds`
- обеспечить корректный цикл ожидания и повторного запуска

Результат этапа:

- daemon может непрерывно работать в foreground
- один цикл не ломает следующие циклы

## Этап 8. Runtime-артефакты и диагностика

Цель:

- использовать `.git/ai-teamlead/` для lock и временных артефактов
- добавить минимально достаточную наблюдаемость
- вести durable session-артефакты и журнал нормализованных действий оператора

Подэтапы:

1. создать layout директорий
2. реализовать `session.json`
3. реализовать `issues/<issue_number>.json`
4. реализовать `questions.json`
5. реализовать `analysis-plan.json`
6. реализовать append-only `operator-events.jsonl`

Результат этапа:

- есть repo-local runtime directory
- есть понятные сообщения stdout/stderr
- есть минимальные диагностические следы запуска

## Этап 9. Smoke verification

Цель:

- пройти основные сценарии из `03-how-we-verify.md`

Минимальный набор:

- daemon стартует в repo с `ai-teamlead.yml`
- `poll` берет issue из `Backlog`
- при нескольких issues `poll` берет минимальный issue number
- `run` запускает допустимую issue
- `run` не запускает issue из waiting-статуса без нужных session-артефактов
- недопустимый `run` корректно отклоняется
- два разных репозитория не мешают друг другу

## Критерий завершения MVP

MVP можно считать реализованным, если:

- foreground daemon запускается в одном репозитории
- `ai-teamlead.yml` читается и валидируется
- `gh` integration работает для чтения и смены статусов
- `poll` и `run` работают по контракту
- `issue-analysis-flow` запускается через `zellij`
- диагностика достаточна для ручного использования без отдельного status API

## Что делать после MVP

- добавить дополнительные CLI-команды при реальной необходимости
- рассмотреть отдельный operator-facing status interface
- рассмотреть режим запуска через `systemd`
- рассмотреть native GitHub API layer при росте сложности
