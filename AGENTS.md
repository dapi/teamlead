# AGENTS.md

## Язык

- Все проектные инструкции, спецификации, SSOT-документы, ADR, шаблоны и
  описания в этом репозитории должны писаться на русском языке.
- Агент должен общаться с пользователем на русском языке.
- Английский допустим только для имен команд, статусов, API-полей, имен файлов
  и иных технических идентификаторов, которые должны оставаться в оригинале.

## Документы

- `README.md` описывает продукт, границы MVP и основные архитектурные решения.
- `docs/code-quality.md` задает quality bar и требования к тестам.
- `docs/documentation-structure.md` задает обязательную структуру документации
  проекта.
- `docs/documentation-process.md` задает процесс работы с документацией.
- `docs/issue-analysis-flow.md` является отдельным SSOT для flow анализа issue.
- Значимые решения по продукту, архитектуре, данным, интерфейсам и runtime
  должны оформляться отдельными ADR с номером.
- Существенные изменения flow сначала фиксируются в SSOT, и только потом
  реализуются в коде.

## Project-Local Agent Assets

- `.ai-teamlead/` содержит project-local flow, launcher и repo-local контракты
  самого `ai-teamlead`.
- `.claude/` используется для Claude-specific project assets.
- `.codex/` используется в этом проекте как repo-level convention для
  Codex-specific assets.
- `AGENTS.md` остается основным repo-level instruction contract для Codex и не
  заменяется каталогом `.codex/`.

## AURA

- `AURA.md` является обязательным project-local приложением к `AGENTS.md`.
- `AURA.md` хранится в репозитории для удобного доступа агента, но описывает не
  историю проекта, а личные устойчивые подходы разработчика к проектированию и
  разработке информационных систем, работе с документацией, декомпозиции,
  архитектурным решениям и инженерному качеству.
- `AURA.md` должен учитываться при любой работе агента в этом репозитории.
- Приоритет источников такой:
  1. прямой запрос пользователя
  2. `AGENTS.md`
  3. `AURA.md`
  4. `README.md`
  5. SSOT-документы
  6. ADR
- `AURA.md` задает высокоуровневые инженерные подходы, эвристики и критерии
  качества мышления.
- `README.md`, SSOT-документы и ADR фиксируют project-specific контракты,
  факты, статусы, пути, конфигурацию, runtime-поведение и другие предметные
  договоренности конкретного репозитория.
- Агент должен интерпретировать и применять project-specific документы через
  призму `AURA.md`, если это не приводит к прямому разрушению зафиксированного
  контракта проекта.
- Если между `AURA.md` и project-specific документом возникает прямое
  противоречие, агент должен явно указать на конфликт и не скрывать его
  молчаливым компромиссом.

## Структура документации

- Значимые фичи и подсистемы должны документироваться по трем осям:
  `Что строим`, `Как строим`, `Как проверяем`.
- Для новых фич нужно использовать единый каркас из
  `docs/templates/feature-spec-template.md`.
- Feature-документы должны иметь номер.
- Если feature-документ становится слишком большим, он должен выноситься в
  отдельную директорию вида `docs/features/XXXX-title/` с набором связанных
  документов.
- Если у фичи есть сложный flow или state machine, для нее должен быть выделен
  отдельный SSOT-документ.

## Flow анализа

- `issue-analysis-flow` развивается как отдельная живая спецификация.
- У flow должны быть явно описаны статусы, переходы и журнал изменений.
- Вопросы по задаче задаются пользователю в агентской сессии, а не в
  комментариях GitHub issue.
