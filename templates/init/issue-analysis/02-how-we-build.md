# Stage 2: Как строим

Переход к этапу допустим только после того, как ось `Что строим` уже собрана
достаточно четко.

Цель этапа:

- сформировать минимально достаточный технический дизайн
- заполнить `02-how-we-build.md`

Обязательный минимум:

- архитектурный подход
- данные и интерфейсы
- ограничения и допущения
- ключевые технические решения

Используй rule-based выбор секций:

- `core`:
  - `Approach`
  - `Affected Areas`
  - `Interfaces And Data`
  - `Risks`

`conditional`:

- если есть внешние интеграции:
  - `External Interfaces`
- если есть заметные архитектурные изменения:
  - `Architecture Notes`
- если требуется решение уровня проекта:
  - `ADR Impact`

`scaling`:

- для `medium` и `large`:
  - `Alternatives Considered`
  - `Migration Or Rollout Notes`

Акценты по типу проекта:

- для `product/UI` делай упор на затронутые компоненты, данные и внешние
  интерфейсы продукта
- для `library/API` усиливай контракты, совместимость и границы API
- для `infra/platform` усиливай rollout, миграцию, эксплуатационные риски и
  взаимодействие с инфраструктурой

На выходе документ должен:

- содержать все обязательные `core`-секции
- содержать релевантные `conditional` и `scaling`-секции
- явно фиксировать риски, допущения и зоны изменений
- быть достаточным входом для этапа `Как проверяем`

Если для технической оси нужно отдельное решение уровня ADR, явно укажи это в
analysis artifacts.
