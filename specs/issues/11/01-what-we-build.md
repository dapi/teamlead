# Issue 11: Что строим

Статус: approved
Последнее обновление: 2026-03-15
Approved By: dapi
Approved At: 2026-03-15T11:52:35+03:00

## Problem

Сейчас ручной запуск `ai-teamlead run <issue>` дает слишком бедные ошибки в
ситуациях, которые для оператора являются ожидаемыми и часто могут быть
исправлены автоматически:

- issue существует, но не добавлена в configured GitHub Project;
- issue есть в проекте, но ее статус не допускает повторный `run`;
- issue уже закрыта;
- пользователь передал номер или URL несуществующей issue.

Текущее поведение смешивает разные причины отказа в две неинформативные
формулировки:

- `issue #N is not linked to the project`
- `run denied for issue #N: status is not a valid run entry point`

Такая диагностика не говорит оператору:

- какой проект ожидается;
- существует ли issue вообще;
- открыта ли она;
- какой у нее текущий статус;
- какие статусы допустимы для stage-aware `run`;
- пыталась ли система что-то исправить сама;
- какое следующее действие нужно сделать вручную, если автоисправление
  невозможно.

## Who Is It For

Изменение нужно двум ролям:

- оператору репозитория, который вручную запускает `ai-teamlead run <issue>` и
  ожидает быстро понять причину отказа и следующий шаг;
- разработчику `ai-teamlead`, которому нужен стабильный и тестируемый контракт
  user-facing ошибок без дублирования ad-hoc форматирования по коду.

## Outcome

После изменения пользователь получает `auto-remediation first` поведение для
ручного `run`:

- если issue не найдена, сообщение явно говорит, что номер или URL не
  разрешились в существующую issue текущего репозитория, потому что эту
  проблему нельзя исправить автоматически;
- если issue закрыта, сообщение сообщает текущее состояние и объясняет, что
  `run` работает только для открытых issue, потому что безопасного
  auto-reopen-path у команды нет;
- если issue не входит в configured проект, система пытается сама добавить ее в
  target project и установить стартовый status, после чего информирует
  пользователя об успешном исправлении или понятном blocker'е;
- если статус не подходит для запуска, система пытается сама перевести issue в
  корректный entry status для текущего flow и либо продолжает `run`, либо
  объясняет, почему не смогла сделать это безопасно.

## Scope

В scope первой версии входят:

- улучшение operator-facing поведения в ручном пути `run <issue>`;
- явное различение случаев `not found`, `closed`, `not in project`,
  `invalid entry status`;
- автоматическое добавление открытой issue в configured GitHub Project, если
  issue еще не прикреплена к нему;
- автоматическая установка стартового project status после такого добавления;
- автоматическая нормализация project status, если для него есть однозначный и
  безопасный remediation-path;
- включение в сообщение project context из `settings.yml`;
- включение в сообщение текущего статуса и списка допустимых статусов там, где
  автоисправление не удалось;
- вынос форматирования отказов и remediation-решений в отдельный доменный
  контракт;
- unit-тесты на формирование сообщений и integration coverage на CLI-path.

## Non-Goals

В эту задачу не входят:

- изменение разрешенных статусов для `run`;
- возврат к analysis-only модели `run`;
- автоматическое переоткрытие закрытой issue;
- произвольное изменение GitHub Project status без детерминированного правила,
  какой target status является корректным;
- локализационная платформа или многоязычный runtime beyond конкретных
  сообщений об ошибках.

## Constraints And Assumptions

- Канонический источник истины о stage и допустимых переходах остается в
  GitHub Project status, как зафиксировано в
  [../../../docs/issue-analysis-flow.md](../../../docs/issue-analysis-flow.md),
  [../../../docs/issue-implementation-flow.md](../../../docs/issue-implementation-flow.md)
  и
  [../../../docs/adr/0024-stage-aware-run-dispatch.md](../../../docs/adr/0024-stage-aware-run-dispatch.md).
- `run <issue>` должен оставаться stage-aware entrypoint и принимать статусы
  implementation lifecycle, которые уже разрешены проектом.
- Автоисправление допустимо только там, где у системы есть однозначный и
  безопасный remediation-path без скрытой магии.
- После успешного автоисправления пользователь должен явно увидеть, что именно
  было исправлено: добавление в project, смена status или другой системный шаг.
- Если автоисправление невозможно, сообщение должно оставаться user-facing, но
  не скрывать технический факт, который нужен для ручного исправления:
  `project_id`, current status, issue state, issue URL.
- Для различения `not in project` и `closed/not found` одного `ProjectSnapshot`
  недостаточно; реализация может потребовать отдельный GitHub lookup вне
  project snapshot.
- Сообщения должны оставаться пригодными для детерминированного тестирования и
  не зависеть от нестабильного форматирования внешнего CLI.

## Observed Behavior

По текущему коду:

- `run_manual_run` сначала грузит `ProjectSnapshot`;
- затем ищет issue только среди `snapshot.items` текущего repo;
- если item не найден, немедленно возвращает
  `issue #N is not linked to the project`;
- если item найден, но `decide_run_stage()` вернул отказ, пользователь видит
  только `status is not a valid run entry point`.

Из-за этого оператор не может отличить:

- закрытую issue вне snapshot;
- несуществующую issue;
- issue другого проекта;
- issue в проекте с неподходящим статусом.

## Expected Behavior

Ручной `run <issue>` должен сначала установить реальный статус issue в
репозитории, затем попытаться исправить проблему автоматически, и только после
этого выбирать одно из человеко-понятных объяснений:

1. issue не существует и потому не может быть исправлена автоматически;
2. issue существует, но закрыта, и команда не переоткрывает ее сама;
3. issue открыта, но не связана с target project, поэтому система пытается
   сама добавить ее в project и установить стартовый status;
4. issue открыта и связана с project, но ее project status не допускает `run`,
   поэтому система пытается сама нормализовать status к корректному entrypoint.

Для каждого случая сообщение должно содержать либо результат автоисправления,
либо минимальный набор данных, достаточный для ручного исправления без чтения
исходников.

## Impact

Без этой задачи ручной операторский цикл остается хрупким:

- пользователь тратит время на догадку, что именно сломано;
- системе приходится эскалировать на пользователя даже те проблемы, которые она
  могла бы исправить сама;
- приходится читать код или логи вместо явной инструкции;
- пример из issue про `Ready for Implementation` может привести к неверной
  реализации, если не учитывать текущий stage-aware контракт;
- отсутствие явного доменного формата ошибок и remediation-path затрудняет
  unit-test coverage и создает риск повторного drift между ветками отказов.
